Configuración técnica de la app
Ajustes en app.json antes de buildear
Verificar que com.fitcore.app no esté en uso en Play Store
Buscar el package name en Google Play. Si ya existe (aunque sea de otro dev), hay que cambiarlo en app.json antes del primer build
Agregar versionCode en app.json
"android": { "package": "com.fitcore.app", "versionCode": 1, "adaptiveIcon": { ... } }
Declarar permisos de Android en app.json
"android": { "permissions": [ "CAMERA", "READ_MEDIA_IMAGES", "INTERNET", "VIBRATE" ] }
Declarar solo los que realmente se usan. Permisos extra pueden hacer que Google rechace la app
Verificar plugins en app.json
"plugins": [ "expo-router", "expo-font", "expo-camera", ["expo-image-picker", { "photosPermission": "FITCORE necesita acceso a tus fotos para analizar comidas." }] ]
Testear en dispositivo físico Android con el build de preview
eas build --platform android --profile preview
Instalar el .apk generado en un celular real (no emulador) y recorrer todos los flujos

PRIMEROBuild Android de producción (.aab)
eas build --platform android --profile production
EAS genera y guarda el keystore automáticamente. Guardar una copia de seguridad del keystore — si lo perdés no podés actualizar la app
Descargar y guardar el .aab generado
El link de descarga aparece en expo.dev/accounts/franmezher o en la terminal al terminar el build
Guardar copia de seguridad del keystore
eas credentials
Descargarlo y guardarlo en un lugar seguro (Google Drive, 1Password, etc.). Sin el keystore no podés publicar updates

6
Post-lanzamiento
Mantener la app viva después del primer deploy
Configurar monitoreo de errores en producción
Sentry (sentry.io) o Firebase Crashlytics. Fundamental para saber si algo se rompe en usuarios reales
Configurar EAS Update para OTA updates
eas update --branch production --message "Fix nutrición modal"
Permite enviar cambios de JS sin generar un nuevo build ni pasar por revisión de Play Store
Responder reviews en Play Store
Afecta el ranking. Responder tanto las negativas como las positivas mejora la visibilidad
Para cada update futuro: incrementar versionCode
// app.json "version": "1.0.1", "android": { "versionCode": 2 } // Luego: eas build --platform android --profile production
El versionCode siempre tiene que ser mayor al anterior. Play Store rechaza builds con el mismo versionCode
